# OWASP Security Evidence

This document records how the Helpdesk Ticket System protects against three OWASP Top 10 vulnerabilities, with step-by-step instructions for manually demonstrating each protection and guidance on what evidence to capture for a written report.

---

## 1. A01:2021 — Broken Access Control

### What the vulnerability is

Broken Access Control occurs when users can act outside their intended permissions — accessing other users' data, reaching admin-only pages, or modifying resources they do not own. It is the most common web application vulnerability category in the OWASP 2021 list.

### How this app protects against it

**Route-level role enforcement** via reusable decorators:

```python
# utils/security.py
def admin_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if current_user.role != 'Admin':
            current_app.logger.warning(
                '[SECURITY] Non-admin accessed %s: user=%s ip=%s',
                request.url, current_user.username, request.remote_addr
            )
            abort(403)
        return f(*args, **kwargs)
    return decorated
```

Applied to every admin route:

```python
# routes/admin.py
@admin_bp.route('/admin/dashboard', methods=['GET', 'POST'])
@login_required
@admin_required
def admin_dashboard():
    ...
```

**Object-level ownership check** before any ticket modification:

```python
# routes/user.py — update action
if ticket.user_id != current_user.id:
    current_app.logger.warning(
        "[SECURITY] User attempted to update another user's ticket: "
        'user=%s ticket_id=%d owner_id=%d ip=%s', ...
    )
    abort(403)
```

**Status change silently ignored for users** — even if a user submits a `status` field, the route discards it server-side without relying on the client to omit it.

**Unauthenticated users** are redirected to the home page by Flask-Login's `@login_required` decorator. The login view is set to `'main.home'`.

### Manual demonstration

**Scenario A — User cannot reach the admin dashboard**

1. Start the app: `python app.py`
2. Register a **User** account at `http://127.0.0.1:5000/choose_role/register`
3. Log in as that user
4. While logged in, navigate directly to `http://127.0.0.1:5000/admin/dashboard`
5. **Expected result:** The custom 403 "Access Denied" page is displayed immediately
6. Check `logs/helpdesk.log` — you should see a line like:
   ```
   2025-01-01 12:00:00 WARNING  [SECURITY] Non-admin accessed http://127.0.0.1:5000/admin/dashboard: user=testuser ip=127.0.0.1
   ```

**Scenario B — User cannot modify another user's ticket**

1. Register two separate User accounts (UserA and UserB)
2. Log in as UserA and create a ticket — note its ID from the URL or dashboard
3. Log out and log in as UserB
4. Open browser DevTools → Network tab
5. On UserB's dashboard, submit the update form for any of UserB's own tickets to capture the raw POST request
6. Use a tool such as **curl**, **Postman**, or **Burp Suite** to replay that POST request, replacing `ticket_id` with UserA's ticket ID:
   ```bash
   curl -X POST http://127.0.0.1:5000/user/dashboard \
     -b "session=<UserB session cookie>" \
     -d "action=update&ticket_id=<UserA ticket ID>&title=Hacked"
   ```
7. **Expected result:** HTTP 403 is returned and the ticket title is unchanged in the database

## 2. CSRF — Cross-Site Request Forgery

> **Note on OWASP 2021 placement:** CSRF was removed from the OWASP Top 10 in the 2021 edition because modern frameworks handle it by default. It remains a critical web vulnerability. Flask-WTF's `CSRFProtect` is used here as an explicit, framework-level control, which demonstrates best-practice implementation.

### What the vulnerability is

CSRF tricks an authenticated user's browser into submitting a forged request to a trusted site — for example, clicking a link on an attacker-controlled page that silently deletes a ticket using the victim's session cookie.

### How this app protects against it

**Global CSRF protection** is enabled in `app.py` for every POST request application-wide:

```python
# app.py
csrf = CSRFProtect(app)
```

Every form template includes a CSRF token hidden field generated by Flask-WTF:

```html
<!-- Inside every <form> tag -->
{{ form.hidden_tag() }}
```

For dashboard forms that share a page (multiple forms per ticket), the CSRF token is injected using:

```html
{{ comment_form.hidden_tag() }}
```

When Flask-WTF validates a POST request without a matching token (or with a tampered token), it returns **HTTP 400 Bad Request**. The 400 error handler logs the event:

```python
# app.py
@app.errorhandler(400)
def bad_request(_):
    app.logger.warning(
        '[SECURITY] 400 Bad Request (possible CSRF): url=%s user=%s ip=%s',
        request.url, actor, request.remote_addr
    )
    return render_template('errors/400.html'), 400
```

### Manual demonstration

**Scenario — Forged cross-origin POST is rejected**

1. Start the app: `python app.py`
2. Open a **second terminal** and run the following curl command (no CSRF token, simulating a cross-origin forged request):
   ```bash
   curl -X POST http://127.0.0.1:5000/login/user \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "username=testuser&password=TestPass1!"
   ```
3. **Expected result:** The server responds with `HTTP/1.1 400 BAD REQUEST` and the body contains "Bad Request"
4. Check `logs/helpdesk.log`:
   ```
   2025-01-01 12:00:00 WARNING  [SECURITY] 400 Bad Request (possible CSRF): url=http://127.0.0.1:5000/login/user user=anonymous ip=127.0.0.1
   ```

**Scenario B — Automated test proof**

The automated test suite demonstrates this without running the server:

```bash
pytest tests/test_errors.py::test_csrf_protection_returns_400 -v
```

The test uses `CSRFConfig` (CSRF enabled) and posts without a token, asserting status 400:

```python
# tests/test_errors.py
def test_csrf_protection_returns_400(csrf_client):
    response = csrf_client.post('/login/user', data={
        'username': 'testuser', 'password': 'TestPass1!'
    })
    assert response.status_code == 400
    assert b'Bad Request' in response.data
```

## 3. A07:2021 — Identification and Authentication Failures

### What the vulnerability is

Weak authentication allows attackers to guess, crack, or bypass credentials. Common issues include weak passwords, no brute-force protection, unsafe credential storage, and unrestricted account creation.

### How this app protects against it

**Strong password policy** enforced by a custom WTForms validator in `forms.py`:

```python
def strong_password(form, field):
    p = field.data or ''
    missing = []
    if not re.search(r'[A-Z]', p): missing.append('one uppercase letter')
    if not re.search(r'[a-z]', p): missing.append('one lowercase letter')
    if not re.search(r'[0-9]', p): missing.append('one digit (0-9)')
    if not re.search(r'[!@#$%^&*()\-_=+\[\]{};:\'",.<>?/\\|`~]', p):
        missing.append('one special character (!@#$%^ etc.)')
    if missing:
        raise ValidationError('Password must contain at least: ' + ', '.join(missing) + '.')
```

Applied to all registration forms (minimum 8 characters + all four character classes).

**Password hashing** — passwords are never stored in plain text:

```python
# routes/auth.py
password=generate_password_hash(form.password.data)  # PBKDF2-HMAC-SHA256
```

Verified on login with `check_password_hash()`. Passwords are never logged.

**Admin registration gated behind a secret code:**

```python
# routes/auth.py
reg_code = current_app.config.get('ADMIN_REGISTRATION_CODE', '')
if not reg_code:
    flash('Admin registration is disabled.', 'danger')
    return redirect(url_for('main.home'))
if form.registration_code.data != reg_code:
    current_app.logger.warning(
        '[AUTH] Admin registration with invalid code: attempted_user=%s ip=%s', ...
    )
    flash('Invalid registration code.', 'danger')
```

**Failed login events are logged** with the attempted username and IP address (but never the password):

```python
# routes/auth.py
current_app.logger.warning(
    '[AUTH] Admin login failure: attempted_user=%s ip=%s',
    form.username.data, request.remote_addr
)
```

**Role separation at the query level** — the login query explicitly filters by role, so an admin account cannot be used on the user login page and vice versa:

```python
# Admin login endpoint
user = User.query.filter_by(username=form.username.data, role='Admin').first()

# User login endpoint
user = User.query.filter_by(username=form.username.data, role='User').first()
```

### Manual demonstration

**Scenario A — Weak password is rejected at registration**

1. Go to `http://127.0.0.1:5000/choose_role/register` → User
2. Fill in a username and valid email
3. Enter a weak password — try each of the following in turn:
   - `password` (no uppercase, no digit, no special)
   - `Password1` (no special character)
   - `Pass1!` (fewer than 8 characters)
4. **Expected result:** The form re-renders with an inline error message such as "Password must contain at least: one uppercase letter, one digit (0-9), one special character..."
5. Only a password meeting all four rules (e.g. `StrongPass1!`) allows registration to proceed

**Scenario B — Admin registration code gate**

1. Ensure `ADMIN_REGISTRATION_CODE` is either empty or set to a known value in `.env`
2. Navigate to `http://127.0.0.1:5000/register/admin`
3. Complete the form with the **wrong** code
4. **Expected result:** "Invalid registration code." flash message; no account is created
5. If `ADMIN_REGISTRATION_CODE` is empty in `.env`, the endpoint redirects immediately to the home page without showing the form at all
6. Check `logs/helpdesk.log` for: `[AUTH] Admin registration with invalid code: attempted_user=... ip=...`

**Scenario C — Wrong role cannot log in**

1. Register a User account (e.g. username `testuser`)
2. Navigate directly to `http://127.0.0.1:5000/login/admin` and attempt to log in with `testuser`'s credentials
3. **Expected result:** "Invalid admin credentials." — the admin login query filters by `role='Admin'` so the User account is not found

## Security Headers (Additional Hardening)

In addition to the three OWASP categories above, the following HTTP response headers are set on every response via an `@app.after_request` hook in `app.py`:

| Header | Value | Protection |
|---|---|---|
| `X-Content-Type-Options` | `nosniff` | Prevents browsers from MIME-sniffing a response away from the declared content type (A05 Security Misconfiguration) |
| `X-Frame-Options` | `DENY` | Prevents the site from being embedded in an `<iframe>`, blocking clickjacking attacks |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Limits the `Referer` header to origin-only for cross-origin requests, reducing information leakage |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | Explicitly denies access to device APIs that this app never needs |
| `Content-Security-Policy` | See below | Restricts which scripts and styles can execute in the browser |

**Content-Security-Policy breakdown:**

```
default-src 'self'                           ← all resources default to same origin
script-src 'self' https://cdn.jsdelivr.net  ← Bootstrap JS from CDN only; no inline scripts
style-src 'self' 'unsafe-inline'            ← inline style="" attributes are present in
         https://cdn.jsdelivr.net           ←  templates; 'unsafe-inline' is the known
                                             ←  trade-off until styles are moved to a .css file
img-src 'self' data:                        ← allows data: URIs (Bootstrap uses these)
font-src 'self' https://cdn.jsdelivr.net    ← Bootstrap icon fonts if added later
frame-ancestors 'none'                      ← no embedding (CSP complement to X-Frame-Options)
base-uri 'self'                             ← prevents <base> tag injection
form-action 'self'                          ← forms can only POST to the same origin
```

**To verify headers are present**, run:

```bash
curl -I http://127.0.0.1:5000/
```

Or run the automated header tests:

```bash
pytest tests/test_security_headers.py -v
```

All 7 header tests pass.
